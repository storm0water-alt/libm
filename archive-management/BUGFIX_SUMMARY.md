# Bug Fix Summary: 日期范围查询功能修复

## 问题描述

档案管理中的日期范围查询无法正确查询到对应日期的档案记录。根本原因是：
1. `Archive` 表的 `date` 字段是 `String` 类型
2. CSV 导入时直接存储原始日期值，没有进行格式标准化
3. 数据库中存在多种日期格式（如 "2024-12-31", "20210127", "2011", "2021年01月27日" 等）
4. 字符串比较无法正确处理不同格式的日期

## 修复方案

### 1. 创建日期解析工具函数 (`lib/utils/date.ts`)

新增了 `parseAndNormalizeDate()` 函数，能够解析多种日期格式并统一转换为 `YYYY-MM-DD` 格式：

**支持的日期格式：**
- `YYYY-MM-DD` (2024-12-31)
- `YYYYMMDD` (20210127)
- `YYYY` (2011) - 自动补全为 2011-01-01
- `YYYY年MM月DD日` (2021年01月27日)
- `YYYY年MM月DD` (2021年01月27)
- `YYYY年M月D日` (2021年1月27日)
- `YYYY年M月D` (2021年1月27)
- `YYYY/MM/DD` (2021/01/27)
- `YYYY.MM.DD` (2021.01.27)

**功能特性：**
- 验证日期有效性（年份范围 1900-2100，月份 1-12，日期根据月份验证）
- 对无效日期返回空字符串并记录警告
- 处理 null、undefined 和空字符串

### 2. 修改 CSV 导入逻辑 (`services/csv-import.service.ts`)

在 CSV 导入时，自动将日期字段标准化：
- 调用 `parseAndNormalizeDate()` 函数处理日期值
- 如果解析成功，存储标准化后的日期（YYYY-MM-DD）
- 如果解析失败，保留原始值并记录警告

### 3. 创建数据库迁移脚本 (`scripts/normalize-dates.ts`)

提供了一个迁移脚本来修复现有数据库中的日期格式：

**使用方法：**
```bash
npx ts-node scripts/normalize-dates.ts
```

**功能：**
- 扫描所有档案记录
- 将各种日期格式统一转换为 YYYY-MM-DD
- 详细的日志输出和统计信息
- 错误处理和验证

### 4. 完整的测试覆盖 (`__tests__/utils/date.test.ts`)

创建了全面的单元测试，覆盖：
- 各种日期格式的解析
- 边界情况（闰年、无效日期等）
- 空值和无效值处理

**测试结果：** ✅ 所有 9 个测试通过

## 技术实现细节

### 为什么使用字符串比较而不是日期类型？

1. **向后兼容性**：保留现有的 String 类型字段，避免破坏性变更
2. **格式统一**：YYYY-MM-DD 格式的字符串可以进行正确的字典序比较
3. **性能考虑**：不需要修改数据库 schema 或进行复杂的数据迁移

### 日期范围查询原理

现在所有日期都统一为 `YYYY-MM-DD` 格式后，字符串比较可以正确工作：

```typescript
// 查询 2021-01-01 到 2021-12-31 之间的日期
where.date = {
  gte: "2021-01-01",  // 大于等于
  lte: "2021-12-31"   // 小于等于
};
```

字符串的字典序比较确保了：
- `"2021-01-27" >= "2021-01-01"` ✅
- `"2021-01-27" <= "2021-12-31"` ✅
- `"20210127"` 不会被匹配（因为所有数据已标准化）
